# 마일스톤 — 학급 투표 앱

순서 고정: Supabase 프로젝트 생성(5단계)은 절대 GitHub 배포(4단계)보다 먼저 오지 않는다.
먼저 배포로 "실제로 돌아간다"는 성취감을 얻고, 마지막에 저장소(Supabase)를 연결한다.

---

## 1단계 — 화면 뼈대
**내용**: index.html에 5개 화면(시작/교사 비번/교사 등록·집계/학생 투표/결과)의 레이아웃만 만든다. 버튼은 있지만 눌러도 아직 아무 데이터도 저장되지 않아도 됨. 밝고 화려한 톤으로 스타일링, 최대 너비 600px 모바일 레이아웃 확인.
**확인 방법**: 브라우저(모바일 너비로 줄여서)에서 index.html을 열어 5개 화면이 각각 예쁘게 보이고, 화면 간 이동(교사/학생 버튼, 뒤로가기 등)이 눈으로 확인된다.

## 2단계 — 로컬 저장 기능 완성 (Supabase 없이)
**내용**: localStorage만으로 전체 기능이 실제로 동작하게 만든다. 교사가 질문을 등록하면 학생 화면에 보이고, 투표하면 기기당 1회 제한이 걸리고, 결과가 실시간(같은 브라우저 내에서는 즉시)으로 갱신된다.
**확인 방법**: 인터넷이 없어도, 브라우저 두 개(또는 시크릿창)를 띄워 교사 화면에서 질문 등록 → 학생 화면에서 투표 → 결과 화면에 반영되는지 직접 눌러서 확인. 같은 창에서 같은 질문에 두 번 투표가 막히는지도 확인.

## 3단계 — Supabase 연동 코드 작성
**내용**: localStorage 대신 Supabase(vote_records 테이블) 읽기/쓰기 코드를 작성한다. config.js에서 URL/key를 읽어오는 구조로 만들고, .gitignore에 config.js를 등록한다. 이 시점엔 실제 Supabase 프로젝트가 아직 없으므로 화면에 "Supabase 설정이 필요해요" 배너가 뜨는 게 정상이다.
**확인 방법**: config.js가 없는 상태에서 index.html을 열었을 때 앱이 에러로 멈추지 않고 안내 배너가 뜨는지 확인. 코드 상에서 Supabase 호출 부분이 올바른 테이블명(vote_records)과 컬럼으로 작성됐는지 코드 리뷰.

## 4단계 — GitHub 배포 (오늘의 유일한 git push)
**내용**: 지금까지 만든 코드를 GitHub 저장소에 올리고, GitHub Pages(또는 원진이 쓰는 배포 방식)로 실제 URL을 만든다. config.js는 .gitignore로 제외되어 있으니 저장소에는 올라가지 않는다.
**확인 방법**: 배포된 URL을 폰으로 직접 열어서 화면이 뜨는지 확인 (이 시점엔 아직 "설정 필요" 배너가 보이는 게 정상). git push가 오늘 처음이자 유일한 push였는지 확인.

## 5단계 — Supabase 프로젝트 생성 및 연결
**내용**: 이제 실제 Supabase 프로젝트를 만들고, vote_records 테이블을 생성한 뒤, URL/anon key를 config.js에 채워 넣는다. 로컬(또는 배포된 사이트)에서 config.js를 반영해 실제 저장이 되는지 확인한다.
**확인 방법**: 배포된 URL에서 "설정 필요" 배너가 사라지고, 교사 화면에서 질문을 등록 → 학생 화면(다른 기기/폰)에서 투표 → 실시간으로 결과가 갱신되는지 실제 폰 2대 이상으로 테스트. Supabase 대시보드에서 vote_records 테이블에 행이 쌓이는지 직접 확인.
